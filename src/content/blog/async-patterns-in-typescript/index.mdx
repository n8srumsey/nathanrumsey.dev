TypeScript's async primitives compose in ways that aren't obvious from the documentation. Here are three patterns that come up constantly and when to reach for each.

## Promise.all — Fail Fast

Use `Promise.all` when all operations must succeed and you want to bail immediately on the first failure.

```typescript
async function fetchUserData(userId: string) {
  const [profile, posts, followers] = await Promise.all([
    fetchProfile(userId),
    fetchPosts(userId),
    fetchFollowers(userId),
  ]);
  return { profile, posts, followers };
}
```

If any of the three requests rejects, `Promise.all` rejects immediately and the other in-flight requests are abandoned (though not cancelled — they'll still complete, their results just won't be used). TypeScript infers the tuple type correctly: `[Profile, Post[], User[]]`.

## Promise.allSettled — Collect All Results

Use `Promise.allSettled` when you want all results, succeeded or failed, before continuing.

```typescript
async function syncItems(items: Item[]) {
  const results = await Promise.allSettled(
    items.map(item => uploadItem(item))
  );

  const failed = results
    .filter((r): r is PromiseRejectedResult => r.status === 'rejected')
    .map(r => r.reason);

  if (failed.length > 0) {
    console.error(`${failed.length} items failed to upload`, failed);
  }

  return results.filter(r => r.status === 'fulfilled').length;
}
```

The type narrowing with a type predicate (`(r): r is PromiseRejectedResult`) is necessary because `allSettled` returns `Array<PromiseFulfilledResult<T> | PromiseRejectedResult>` and TypeScript can't narrow on `.status` without help in a `.filter()` callback.

## Async Generators — Streaming Results

Use async generators when the producer naturally yields results over time and the consumer wants to process them as they arrive, without buffering everything in memory first.

```typescript
async function* paginate<T>(
  fetchPage: (cursor: string | null) => Promise<{ items: T[]; nextCursor: string | null }>
): AsyncGenerator<T> {
  let cursor: string | null = null;
  do {
    const { items, nextCursor } = await fetchPage(cursor);
    for (const item of items) {
      yield item;
    }
    cursor = nextCursor;
  } while (cursor !== null);
}

// Consuming it
for await (const record of paginate(fetchRecordsPage)) {
  await processRecord(record);
}
```

The caller sees a flat stream of `T` values; the pagination logic is fully encapsulated. The `for await...of` loop handles backpressure implicitly — the generator suspends until the consumer is ready for the next item.

## Choosing

| Scenario | Pattern |
|---|---|
| All must succeed, fail fast | `Promise.all` |
| Collect all outcomes, partial success OK | `Promise.allSettled` |
| Unbounded or large result set | Async generator |
| Sequential with early exit | `for await` with `break` |

The most common mistake is using `Promise.all` where `Promise.allSettled` is needed — batch operations almost always need the "collect all results" behavior, not "fail on first error."
