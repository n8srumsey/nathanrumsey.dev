Most developers know that indexes make queries faster. Fewer know *why*, or when they don't help. Understanding the data structure underneath changes how you design schemas.

## The B-Tree

Nearly every relational database uses a B-tree (or B+ tree variant) as the default index structure. A B-tree is a self-balancing tree where every node holds multiple keys and every leaf is at the same depth. For a table with millions of rows, a B-tree index keeps lookups to O(log n) instead of O(n).

![B-tree index structure](https://picsum.photos/seed/btree/800/400)

The leaf nodes in a B+ tree also maintain a linked list between siblings. This is what makes range scans efficient — once you find the starting key, you follow the leaf chain without returning to the root.

## Creating an Index

```sql
-- Single-column index
CREATE INDEX idx_users_email ON users (email);

-- Composite index — column order matters
CREATE INDEX idx_orders_user_date ON orders (user_id, created_at);

-- Partial index — only indexes rows matching a condition
CREATE INDEX idx_active_users ON users (email) WHERE deleted_at IS NULL;
```

The composite index `(user_id, created_at)` can satisfy queries that filter on `user_id` alone, or on both `user_id` and `created_at` together. It cannot satisfy a query that filters only on `created_at` — that would require a full index scan.

## Reading EXPLAIN

```sql
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE user_id = 42
  AND created_at > '2026-01-01';
```

```
Index Scan using idx_orders_user_date on orders
  Index Cond: ((user_id = 42) AND (created_at > '2026-01-01'))
  Rows Removed by Filter: 0
  Planning Time: 0.1 ms
  Execution Time: 0.4 ms
```

`Index Scan` is good. `Seq Scan` on a large table is usually a sign something is missing. `Bitmap Heap Scan` appears when the planner decides it's cheaper to collect matching row IDs from the index first, then fetch them in heap order — common for queries returning a large fraction of a table.

## When Indexes Hurt

Indexes aren't free. Every write to the table requires maintaining every index on it. A table with ten indexes is fast to read and slow to write. For tables that are insert-heavy and read rarely (event logs, audit trails), indexes are often counterproductive. For OLAP workloads that run full-table aggregations, a columnar store beats a row store with indexes.

The rule: measure before adding. `EXPLAIN ANALYZE` shows actual execution time, not estimates. Run it with and without the index under realistic data volumes.
